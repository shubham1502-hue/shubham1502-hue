# GitHub Profile Strategy

This profile should not read like a code portfolio. It should read like a working founder-operator toolkit.

## The Core Narrative

The profile needs to prove three things quickly:

1. I understand founder problems.
2. I can turn messy operating work into systems.
3. The systems are useful enough for someone else to fork.

The message is not "I know Python" or "I made projects." The message is:

> I build practical operating leverage for founders.

## How A Founder Reads The Profile

A founder usually scans in this order:

1. Name and headline.
2. Pinned repos.
3. README first screen.
4. Repo names and descriptions.
5. One repo README.
6. Whether the repo looks usable.

That means the first screen has to answer:

- What kind of work do you do?
- What problem can I trust you with?
- Which repo should I click first?
- Can this be adapted to my company?

## What Each Repo Is Supposed To Prove

### Founder Weekly Operating Review Agent

Signal: I can turn weekly metrics into founder decisions.

Use this when speaking to founders who are running reviews manually.

### Board Pack / Investor Update Agent

Signal: I understand investor and board communication, not just dashboards.

Use this when speaking to founders preparing updates, fundraising, or board meetings.

### AI GTM Command Center

Signal: I can build founder-led GTM infrastructure without turning outreach into spam.

Use this when speaking to AI/SaaS founders doing outbound, account research, or early sales.

### RevOps Infrastructure Playbook

Signal: I have actually built operating infrastructure inside a founder-led company.

Use this when speaking to founders with CRM, handoff, reporting, or revenue leakage problems.

### Operations KPI Automation

Signal: I can turn ops queues into measurable dashboards.

Use this when speaking to founders with support, onboarding, fulfillment, or SLA issues.

### Collections Strategy Engine

Signal: I can turn a business strategy question into an experiment.

Use this when speaking to fintech, lending, or risk founders.

## How To Explain The Profile In A DM

Do not say:

> Please check my GitHub, I am looking for a role.

Say:

> I have been building a forkable Founder OS toolkit: weekly operating reviews, board updates, GTM research, RevOps templates, and ops dashboards. I thought one repo might be useful for what you are building.

Then mention one repo only.

## DM Rule

Every outbound message should map:

```text
Founder signal -> likely operating pain -> one relevant repo -> small feedback ask
```

Example:

```text
Saw you are hiring for GTM ops after the seed round.
That usually means account research, follow-ups, and CRM hygiene start breaking.
I built a forkable AI GTM Command Center for that workflow.
Open to me sending it for feedback?
```

## What Not To Do

- Do not send all repos at once.
- Do not ask for a job in the first message.
- Do not make GitHub the subject; make the founder's problem the subject.
- Do not over-automate LinkedIn.
- Do not optimize for message volume before message quality.

## Weekly Review Habit

Every week, review:

- Which founders replied?
- Which repo got the most interest?
- Which message had the clearest pain-to-repo mapping?
- Which target segment was too broad?
- Which repo needs one improvement based on feedback?

Then improve one repo or one message before increasing volume.
